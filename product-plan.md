# 产品化规划：工具聚合小程序与微博热搜提醒

## 1. 核心结论

当前项目应先聚焦“微博热搜订阅提醒”这个核心功能闭环，不建议一开始投入过多精力做完整工具聚合平台。

推荐方向：

- 产品外壳可以是“工具聚合小程序”。
- MVP 主功能应是“微博热搜订阅提醒”。
- 计算器、天气、第三方小程序跳转等工具可以作为轻量入口或占位能力。
- 后端主服务继续承担核心业务逻辑。
- 微信云函数只作为微信生态能力的轻量钩子，主要用于发送小程序订阅消息。

核心闭环：

```text
微信登录
-> 创建热搜订阅规则
-> 用户授权订阅消息
-> 主服务定时抓取和匹配
-> 命中后按策略通知用户
-> 用户回到小程序查看命中详情
```

## 2. 产品定位

产品可以定位为：

```text
一个以热点预警为核心的微信工具小程序。
```

短期重点不是“大而全工具箱”，而是：

- 让用户快速订阅关心的热搜词条。
- 在相关话题上榜、爆发、升温时收到提醒。
- 支持微信小程序提醒和外部 Webhook/飞书/企微/钉钉/Telegram 通道。
- 在小程序中查看命中记录、趋势和推送状态。

## 3. 小程序跳转其他小程序

微信小程序可以通过 `wx.navigateToMiniProgram` 跳转到其他小程序。

需要准备：

- 目标小程序 `appId`
- 可选 `path`
- 可选 `extraData`
- 通常由用户点击触发

产品建议：

- 可以做“工具广场”。
- 第三方小程序跳转只作为辅助入口。
- 不要把第三方跳转作为核心壁垒。
- 第三方小程序的可用性、路径和体验不可控。

示例工具广场：

```text
工具广场
- 微博热搜提醒：自研核心功能
- 计算器：轻量自研或跳转第三方
- 天气：轻量自研或跳转第三方
- 汇率换算：轻量自研或跳转第三方
```

## 4. 微信登录与用户体系

微信登录是 P0。

推荐用户模型：

```text
users
- id
- openid
- unionid
- nickname
- avatar
- createdAt
- lastLoginAt
```

登录流程：

```text
小程序 wx.login 获取 code
-> 小程序将 code 发给主服务
-> 主服务调用微信 code2session
-> 主服务拿到 openid
-> 创建或更新用户
-> 主服务返回自己的登录 token
```

建议：

- 不要把核心用户体系完全放在云函数里。
- 主服务应掌握用户、订阅、通道、命中事件和通知策略。
- 云函数只做微信能力代理。

## 5. 小程序订阅消息限制

小程序订阅消息不是无限推送通道。

重要规则：

- 需要用户通过 `wx.requestSubscribeMessage` 授权。
- 一次性订阅通常是“用户授权 1 次，服务端可发送 1 条”。
- 长期订阅有类目限制，通常不适合普通商业工具随意使用。
- 服务端发送前应检查用户是否还有可用授权额度。

因此产品上要设计：

- “开启提醒”授权入口。
- “补充提醒额度”入口。
- 微信订阅消息只用于重要提醒。
- 高频持续通知应更多依赖 Webhook、飞书、企微、钉钉、Telegram。

推荐流程：

```text
用户点击开启提醒
-> 小程序调用 wx.requestSubscribeMessage
-> 用户授权
-> 主服务记录该用户可发送额度
-> 命中热搜时消耗一次额度
-> 云函数调用微信订阅消息 API 发送
```

## 6. 主服务 + 云函数钩子架构

推荐架构：

```text
微信小程序
-> 登录、订阅配置、授权订阅消息、查看命中事件

主服务
-> 用户系统
-> 订阅规则
-> 热搜抓取
-> 命中事件聚合
-> 去重与通知策略
-> Webhook/飞书/企微/钉钉/Telegram 分发
-> 必要时调用云函数

微信云函数
-> 校验主服务签名
-> 调用 cloud.openapi.subscribeMessage.send
-> 返回发送结果
```

这个方案的优点：

- 云函数调用次数只与实际微信订阅消息发送次数相关。
- 抓取频率不会直接放大云函数调用次数。
- 主服务保留核心业务控制权。
- 云函数只承担微信生态相关能力。

错误做法：

```text
每 1 分钟抓取一次
-> 每次都调用云函数
-> 云函数调用量爆炸
```

正确做法：

```text
每 1 分钟抓取一次
-> 主服务本地匹配、聚合、去重
-> 只有真正需要微信提醒时才调用云函数
```

## 7. 命中事件模型

不能把“每次定时任务匹配到”都当成一个新的命中。

否则抓取频率越高，命中次数越膨胀。

推荐拆分：

```text
MatchEvent：命中事件
MatchObservation：命中观测点
```

### 7.1 MatchEvent

表示某个订阅命中了某个话题的一轮生命周期。

字段建议：

```text
match_events
- id
- userId
- subscriptionId
- keyword
- normalizedKeyword
- firstSeenAt
- lastSeenAt
- observedCount
- firstRank
- bestRank
- latestRank
- maxHotValue
- latestHotValue
- latestLabel
- status
- deliveryStatus
```

唯一聚合逻辑：

```text
userId + subscriptionId + normalizedKeyword + activeWindow
```

### 7.2 MatchObservation

表示某次抓取中，这个事件被再次观察到。

字段建议：

```text
match_observations
- id
- eventId
- snapshotId
- observedAt
- rank
- hotValue
- label
```

MVP 可以先不做 `match_observations`，只在 `match_events` 上做 upsert：

```text
首次命中：创建 MatchEvent
重复命中：更新 lastSeenAt、observedCount、latestRank、bestRank、latestHotValue、maxHotValue
```

## 8. 通知策略

通知不应基于每次观测，而应基于事件变化。

默认通知条件：

- 第一次命中。
- 标签升级为“爆”或“沸”。
- 排名进入前 N。
- 热度超过用户设置阈值。
- 热度增长超过一定比例。

普通重复观测不推送。

这样抓取频率变高只会提升趋势精度，不会提升推送噪音。

## 9. P0 优先级

P0 是让核心产品闭环成立。

### P0-1 微信登录

- 小程序 `wx.login`
- 主服务 `code2session`
- 建立 `openid -> userId`
- 返回主服务 token

### P0-2 用户私有数据

需要按用户隔离：

- 订阅规则
- 通道配置
- 命中事件
- 通知日志
- 微信订阅消息额度

热搜快照可以作为全局数据共享。

### P0-3 热搜订阅创建

基础能力：

- 关键词订阅
- 标签过滤：爆、热、新、沸
- 最低热度
- 排除词

### P0-4 命中事件聚合

新增 `match_events`。

不要按任务频率累计“命中次数”。

### P0-5 微信订阅消息授权与发送

- 小程序端申请授权。
- 主服务记录可用发送额度。
- 云函数负责发送。
- 发送成功后扣减额度。

### P0-6 主服务到云函数安全钩子

云函数必须校验主服务请求。

至少包括：

- shared secret
- timestamp
- nonce
- signature
- 防重放校验

### P0-7 通知降噪

默认只通知重要事件，避免刷屏。

## 10. P1 优先级

P1 是提升核心体验。

### P1-1 订阅规则预览

用户配置规则时，立即展示：

- 当前热搜可命中几条
- 命中哪些话题
- 会推送到哪些通道

### P1-2 命中记录列表

小程序中展示：

- 今日新增相关话题
- 正在观察中的话题
- 已通知
- 未通知原因

### P1-3 通知渠道管理

支持并完善：

- 微信小程序订阅消息
- 飞书 Webhook
- 飞书自建应用
- 企业微信
- 钉钉
- Telegram
- 通用 Webhook

### P1-4 抓取状态页

展示：

- 最近抓取时间
- 抓取状态
- 抓取条数
- 失败原因
- 下一次抓取时间

### P1-5 简单工具广场

可以加入：

- 微博热搜提醒
- 天气
- 计算器
- 汇率
- 第三方小程序跳转

## 11. P2 优先级

P2 可以后置。

- 完整工具生态
- 大量第三方小程序跳转
- 团队/租户体系
- 复杂权限
- 报表中心
- 日报周报
- 计费系统
- AI 舆情分析
- 多平台 App

## 12. 推荐数据库表

```text
users
subscriptions
channels
hot_search_snapshots
match_events
match_observations
notification_quota
notification_logs
```

其中：

- `hot_search_snapshots` 是全局数据。
- `subscriptions`、`channels`、`match_events`、`notification_logs` 是用户数据。
- `match_observations` 可以后置。

## 13. 当前项目落地顺序

基于现有 Spring Boot 主服务，建议顺序：

1. 增加微信用户表和微信登录接口。
2. 所有订阅、通道、日志按 `userId` 收口。
3. 新增 `match_events`。
4. 增加小程序端最小页面：登录、订阅列表、新建订阅、命中列表。
5. 新增云函数 `sendSubscribeMessage`。
6. 主服务命中后按通知策略调用云函数。
7. 再做工具广场入口。

## 14. 关键判断

主服务 + 云函数钩子的方案是合理的。

最终职责划分：

```text
主服务：重逻辑、数据、匹配、去重、通知策略
云函数：微信订阅消息发送
小程序：用户入口和授权入口
外部通道：高频持续通知
```

不要把云函数当主后端。

不要把小程序订阅消息当无限推送通道。

不要一开始做大而全工具平台。

先把“微博热搜提醒”打透，再用工具广场承载扩展入口。
